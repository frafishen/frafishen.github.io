NLP 模型探索

主要參考

  1. 李宏毅教授的影片說明
  2. 論文BERT: Pre-training of Deep Bidirectional Transformers for Language Understanding

Contextualized Word Embedding

  • 原本的 word embedding 基本上就是每個word會有一個自己的向量，也就是每個type(字)只會有個token(意思->向量)
  • Contextualized 的技術就是讓同一個type，在句子中，會根據前後的type的token去判斷，應該自己是哪個token

ELMO

  • ELMO 是基於 RNN 的模型
    • 只需要收集一堆的句子，不需要特別去進行標籤
    • 訓練的點：下一個會出現的token是什麼
    • 正反都做一次，就能分析上下文了
    • 因為deep learning，所以會有很多層，所以分析出來，可能也會有很多個token的結果。ELMO的做法就是對每一層有一個參數(相對於超參數，是動態的，因為對於不同的問題，不同層的結果重要程度可能會不同)，對每一層加權，共同決定最後預測的token

BERT

  • encoder of transformer 就是bert的network架構
  • 一般如果要transform一個transformer的話，就需要有一些東西pass過去，像是 translation, summerization之類的，但bert的邏輯，就是直接去transform transformer的encoder就可以了，所以不需要上label，直接丟句子進去就可以了
  • 運作邏輯就是給一串sequence，bert就會吐一串embedding的sequence出來
  • 中文如果利用bert的話，顆粒度用字等級的，會更為恰當，因為用詞做embedding的話，計算的維度會太大

  • BERT怎麼訓練的?

    Masked LM

  • 隨機蓋掉裡面15%的字詞，用[MASK]代替，用克漏字的方式去訓練模型。
  • 每個被[MASK]的地方，就會產生embedding的vector，那這個vector再丟進一個Linear Multi-class classifier，去預測被[MASK]掉的是哪個詞彙。
  • 如果兩個詞彙填在同一個地方沒有違和感，那就代表他們語意差不多，也就表示出來的embedding會很相近

    Next sentence prediction

  • 在兩個句子中間插入一個特殊的token，[SEP]，表示兩個句子的交界
  • 還有一個特殊的token，[CLS]，表示從這裡開始去做classification
    • 從這裡出來的embedding，丟進Linear binary classifier(輸出yes or no)，要判斷的就是，這句話跟其他句話是不是應該要接在一起的。(屬不屬於上下文句子?)
  • bert的內部不是RNN，是一個transformer
    • 內部是self attention，不考慮positional encoding的情況，token放在句首還是尾其實沒有區別
    • self attention跟雙向RNN有同樣能力，可以先偷看過整個sequence，但特別的是輸入的type是同時處理計算token的，不需要一個算完才能算下一個
    • 以前用RNN發表過的paper已經全部被使用self-attention機制洗一輪了… by李宏毅教授

“Masked LM” & “Next sentence prediction” 一起用，效果最好

BERT怎麼用?

Case 1

  • 輸入一堆句子，輸出文章類別
    • 在輸入的最前面加入[CLS]
    • 將輸出的embedding再丟入Linear Classifier (train from scratch)
    • 然後就能預測丟入的句子1, 句子2, 句子3…屬於什麼類型(class)的文章
    • 應用：情緒分析、文本分類

      Case 2

  • 輸入單一句子，輸出每個字詞的類別
    • 在輸入的最前面加入[CLS]
    • 將輸出的embedding再丟入Linear Classifier (每個字詞都會有embedding)
    • 應用：slot filling

      Case 3

  • 輸入兩個句子，輸出判斷類別(成立/不成立/不知道)
    • 輸入：[CLS]sen1[SEP]sen2
    • 應用：Natural Language Inference　-> 根據前提，這個假設成不成立，或是不知道

      Case 4

  • 輸入一篇文章，跟問題，答案如果有出現在文章裡面，就能夠輸出答案
    • 輸入：[CLS]question sequnce[SEP]document
    • 輸出：答案在document中的哪裏到哪裡
    • 文章裡的每個字詞都有embedding
    • 應用：Extraction-based Question Answering(QA), (eg, SQuAD)

NLP 工程師養成術 讀書心得 (1) - 隱馬可夫模型

前面三章主要是在講述何謂NLP，建構一些框架性的觀念，並在第三章中介紹了n元語法模型。
但有個硬傷: n元語法模型是 based o n 已存在之語料庫進行搜索優先序，所以註定無法處理 OOV。
為了解決 OOV 的召回問題，我們會不同方式對新詞(句)進行標註，讓模型認識新詞彙。

n元語法模型: 是一個能夠求一個句子中，最大機率切分的模型。n = 句子的長度理論上可以是無限大。
OOV: out of volcabulary 超出表外的字
召回: 將字詞收錄到字典中

Chapter 4 隱馬可夫模型與序列標註

Hanlp 有提供基於隱馬可夫模型的中文分詞器 HMMSegmemnt
下面是假設雨模型的理論

OOV 沒辦法被”n元語法模型”召回，是因為它的顆粒度是詞彙層級，阿 OOV 就不在詞彙典裡，所以一定要回到字元層級去進行重新認識
這個行為被稱為”序列標註”行為，其中隱馬可夫是最簡單的模型。

馬可夫假設: 每個事件發生的機率，指跟前一個發生的事件有關系。(有點像貝氏)
很多事件串在一起就是馬可夫鏈。
在NLP中，事件 -> 單詞，馬可夫模型 -> 二語法模型。

隱馬可夫鏈的假設與結構

隱馬可夫鏈中有兩種序列: 觀測序列(x)、狀態序列(y)。
由狀態可以由相對應的觀測推得。
隱馬可夫鏈所做的假設就是：
x 外界可見，y 外界不可見
(1) x[i] 會由觀察 y[i] 所推得，但實際上y長什麼樣子我們不一定知道。
(2) y[i] 取決於 y[i-1]。

隱馬可夫模型利用三個要素模擬序列發生的過程
程式實作: p.156，這個是資料及完整的情況下

初始狀態機率向量

第一個字元是哪個字元的機率向量序列，其中序列合為1。

狀態轉移機率矩陣

假設一個字串長度為N，則會產生一個N*N的狀態轉移機率矩陣。
為狀態i到j，所發生的機率。
因為有提到i只會跟i-1有關，所以需要有一個類似前綴的矩陣。

發射機率矩陣

假設觀察x共有M種可能性，而y本身有N種，那決定x和y之後就會產生一個M*N的發射機率矩陣。
形象化: 將y[i]想像成不同的彩彈槍，x[i]為不同的彩彈。根據”發射機率矩陣”可以決定，拔哪支槍y[i]，而不同的槍，不同顏色的子彈也會有不同的比例x[i]。

只要確定三元組，便可以確定隱馬可夫模型了。

樣本生成演算法

考慮長T的樣本序列(字串)，生成過程就是沿著引馬可夫鏈走T步
程式實作: p.158

模型的訓練

對模型來說，
觀測序列(x): 特徵資料，以這些資料去建立模型
狀態序列(y): 標籤資料，標準答案
因此，資料及有沒有標註y，就可以對應到我們應該要建立，監督式學習，或是非監督式學習。

隱馬可夫鏈的參數是上面提到的三要素。
估計的程式實作於 p.159，用於資料集不完整，所以需要訓練模型，以進行推估

p.164~p.170 主要在講狀態向前、搜尋的演算法實作

處理字串匹配問題之前，除了最暴力直接枚舉匹配，複雜度直上O(m * n)，先學學資料結構

Trie 字典樹

以字典數對字串做儲存的話，可以減少儲存空間，並些讓 travel 及 insert 的時間複雜度降到線性

KMP

Hashing Method 用來 access 字串不是不行，但它畢竟是一個隨機演算法，且模運算其實很花時間。
KMP = Knuth-Morris-Pratt algorithm

創建一隻APP

使用Android Studio 進行開發。一開始的初始檔案就超級多，所以後面有好一大端時間是在學習那些檔案、標籤是在做什麼。

基本流程

  • choose your project -> Empity Activity
  • Android Studio 讓開發者有很多種閱讀專案的設定，先用Android
  • 一開始的APP，就是default 為Hello World!
  • 有幾個比較重要的檔案之後會在裡面遊蕩
    • app/src/main/java//MainActivity.kt: 主要寫APP功能的地方，定義物件，設定物件ID，以及寫listener function
    • app/src/main/res/layout/activity_main.xml: 這個就是最底層，一打開畫面的layout，也就是MainActivity.kt 的輸出畫面。在這裡可以使用切分畫面，直接用拉的設計GUI
    • app/src/main/res/drawable: 將所需要的image 都放在這裡
    • app/src/main/res/values/strings.xml: 定義一些重要的字串，比如這隻APP叫什麼
    • app/build.gradle: 設定版本資訊
  • 在debug 或是compile & run 的時候，有兩種方式:
    • 使用AVD(Android Virtual Device)，選擇上面的bar: Tools>Device Manager，create virtual device，選擇Nexus 5 或是Pixel XL，其他就挑自己喜歡的設定。老實說，我下載完之後有點後悔，因為我诶有想到這個虛擬機，下載完之後就直接佔了1.5G，假如手邊真的沒有Android Device再使用這個方法吧。
    • 使用Physical Device，就是手邊如果有實體安卓裝置的話，會方便很多。首先需要打開USB debugging，方法是要先打開裝置的Develpoer options，每個裝置不太一樣，可以查一下。最後就直接接線，就可以用
